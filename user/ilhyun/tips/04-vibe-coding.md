# 04. 바이브 코딩 마인드셋

이 문서가 제일 중요해요.
형이 지금 완벽하게 이해하지 못해도 괜찮은 이유를 설명하는 문서예요.

---

## 바이브 코딩이 뭐예요?

완벽히 이해한 뒤 시작하는 방식이 아니라, **일단 돌아가는 걸 만들고 그걸 뜯어보면서 배우는 방식**이에요.

쉽게 말하면 이거예요.

- 전통 학습: 공부를 오래 하고 나중에 만들기 시작
- 바이브 코딩: 먼저 만들고, 만들면서 질문하고 이해하기

저는 보통 후자를 더 자주 써요.
초보일수록 작은 성공이 빨리 나오기 때문이에요.

---

## 3원칙

### 1. 물어보는 걸 부끄러워하지 말 것

초보 질문은 부끄러운 게 아니에요.
오히려 빨리 물어볼수록 빨리 앞으로 가요.

예시:

```text
이 함수가 뭐 하는지 초보 기준으로 설명해줘
```

---

### 2. 한 번에 하나만

처음부터 큰 게임 하나를 통째로 만들려 하면 거의 반드시 꼬여요.
작게 쪼개야 해요.

좋은 흐름은 이런 거예요.

1. 화면 띄우기
2. 버튼 하나 만들기
3. 버튼 누르면 점수 올라가게 하기
4. 점수 초기화 버튼 추가하기

---

### 3. 망해도 Git이 되돌려줌

이게 마음을 편하게 해줘요.
커밋만 해두면 실험이 덜 무서워져요.

```bash
git add .
git commit -m "실험 전 저장"
```

이러고 나면 바꿔보다가 꼬여도 다시 돌아갈 지점이 생겨요.

---

## 전통 학습 vs 바이브 코딩

| 방식 | 흐름 | 장점 | 단점 |
|------|------|------|------|
| 전통 학습 | 문법 공부 → 예제 풀이 → 프로젝트 | 체계가 잘 잡힘 | 시작이 느리고 쉽게 질림 |
| 바이브 코딩 | 만들기 → 돌려보기 → 질문 → 수정 | 빠르게 손에 잡힘 | 이유를 안 물으면 겉핥기로 끝날 수 있음 |

정답은 하나가 아니에요.
다만 형처럼 "일단 만들어보고 싶다" 타입이면 바이브 코딩이 훨씬 잘 맞아요.

---

## 실제 사례 — 간단한 게임 만들기 타임라인

예를 들어 "버튼 누르면 점수 올라가는 미니 게임"을 만든다고 해볼게요.

| 시간 | 하는 일 |
|------|---------|
| 0분 | Claude에게 `HTML, CSS, JS로 점수 버튼 게임 기본판 만들어줘` 요청 |
| 10분 | 실행해보고 진짜 버튼이 눌리는지 확인 |
| 15분 | `이 함수가 점수를 올리는 부분이야?` 질문 |
| 20분 | 색깔 바꿔달라고 요청 |
| 25분 | `점수 10점 넘으면 축하 문구 띄워줘` 요청 |
| 35분 | 이상한 동작 나오면 에러 이유 질문 |
| 45분 | `git add . && git commit -m "첫 게임 프로토타입"` |

이 흐름의 핵심은 **이해 100%가 아니라, 작동하는 걸 기준으로 질문이 생기게 만드는 것**이에요.

---

## 제일 중요한 습관

AI가 만든 코드를 그냥 넘기지 말고, 꼭 한 번은 **내 머리로 해부**해봐야 해요.

저는 보통 이렇게 물어요.

```text
방금 만든 코드에서 진짜 중요한 부분 3개만 골라서 설명해줘
```

또는:

```text
이 코드에서 내가 직접 손대도 되는 부분이 어디야?
```

이 과정을 거치면 "남이 만든 코드"가 아니라 점점 "내가 이해한 코드"가 돼요.

---

## 초보가 자주 빠지는 함정 3개

### 함정 1. 한 번에 너무 크게 시킴

문제:
```text
2D 온라인 게임 하나 만들어줘
```

극복:
```text
HTML로 시작 화면부터 만들어줘
```

작게 쪼개면 성공률이 훨씬 올라가요.

---

### 함정 2. 돌아가면 끝이라고 생각함

문제:
작동은 하지만 왜 되는지 모르면 다음에 다시 막혀요.

극복:
항상 한 번은 이렇게 물어요.

```text
왜 이렇게 만들었는지 설명해줘
```

---

### 함정 3. 커밋 없이 오래 작업함

문제:
몇 시간 치가 한 번에 날아갈 수 있어요.

극복:
작은 기능 하나 끝날 때마다 저장해요.

```bash
git add .
git commit -m "작은 단위 저장"
```

---

## 한 줄 요약

형은 **완벽히 이해한 뒤 시작하려 하지 말고, 작은 걸 먼저 돌리고, 왜 그렇게 됐는지 물어보는 습관**만 들이면 돼요.
